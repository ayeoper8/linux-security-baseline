# Decisions & accepted limitations register

The audit answer for "why is it like this." Change entries only with a recorded reason.

| # | Decision | Rationale | Compensating controls / revisit trigger |
|---|---|---|---|
| 1 | Scanner account elevates via `sudo su -` with **NOPASSWD** (effectively root) | Qualys requires UID=0 for full detection and runs exactly `sudo su -` non-interactively; command allowlists are not supported by Qualys; a sudo password cannot be typed by the scanner | Key-only auth; private key exists solely in Key Vault; `from=` source restriction to scanner IP; locked password; sudoers permits only `su -`. Mirrors the accepted Windows domain-credential scanning model. |
| 2 | One SSH keypair for the whole estate | One auth record, one vault secret, rotation feasible without per-host records | `from=` restriction makes a leaked key unusable except from the appliance IP; rotation runbook in docs/01 |
| 3 | SSH (22/tcp) open from **any** source in UFW | Flat Class A network, DHCP clients, VPN remote work — source restriction unworkable | sshd hardening; key auth for admin accounts; root login disabled where possible. Revisit if admin VLAN / bastion is introduced. |
| 4 | Stricter than CIS: `hmac-sha1` and `hmac-sha1-etm` MACs removed | CIS 24.04 L1 does **not** remove these, but Qualys QID 38909 flags them | None needed (tightening). Verify scanner appliance still negotiates — tested before rollout. |
| 5 | Auditing deferred: no auditd ruleset, no log forwarding to Sentinel | Sentinel ingestion cost not yet assessed; no current analysis consumer; ops burden mismatched to team | CrowdStrike provides endpoint telemetry/detection on these hosts. Revisit on audit/IACS demand or cost assessment. |
| 6 | CIS implemented as a minimal essentials subset, not full L1 | Windows-majority infra team, no config management; only set-once-static controls included to avoid an unmaintained sprawl | USG/OpenSCAP audit-only runs available at any time for measurement; tailoring/expansion as team capability grows |
| 7 | ~~sysctl network hardening declined~~ **Superseded 12 Jun 2026**: adopted in the M10 hardening pack along with fs module blacklist, core dump restriction, cron/at lockdown, banner | All are static set-once controls — they failed the original exclusion test for deferral | n/a |
| 8 | Root SSH login disabled **conditionally** | Avoid locking admins out of hosts that have no named admin accounts yet | Script verifies a sudo-group user with a populated authorized_keys exists before applying; warns otherwise |
| 9 | sshd changes applied with **reload only**, never restart | Estate is administered over SSH; restart risks dropping the session applying the change | `sshd -t` validation before reload; drop-in removed on validation failure |
| 10 | Unattended-upgrades auto-reboot **off** by default | Reboot policy is an infrastructure decision not yet made | "Reboot required" hosts will be visible in Qualys; decide jointly with infrastructure |
| 11 | `ansible-mgmt` account with **NOPASSWD: ALL** sudo deployed to every host (M9) | Estate has no remote execution capability; key rotation and any baseline change would otherwise mean manually touching 150+ hosts; the baseline rollout is the one free window to enrol every host for future Ansible management | Same controls as #1/#2: key-only, locked password, private key solely in Key Vault, `from=` restricted to control node IP(s). Control node must itself be baseline-hardened and treated as the highest-value Linux asset; flagged to Head of Cyber before rollout. Full `ALL` sudo is required by config management (unlike the scanner's `su -` alias). |
| 12 | Script is strictly non-interactive: decisions via flags (`--role`, `--auto-reboot`, `--admin-user`/`--admin-key`), never prompts | Prompts break cloud-init/first-boot and Ansible execution paths, and push decisions onto the person least placed to make them | Flags are visible in shell history; defaults are the safe option (auto-reboot off, role standard) |
